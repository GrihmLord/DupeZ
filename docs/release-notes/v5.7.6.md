# DupeZ v5.7.6 — Maximum Security Tier 1

**Release date:** 2026-05-27
**Theme:** Close the five highest-ROI residual attack paths from the post-v5.7.5 security review.

This is a security-only release. No new features, no UI changes. If v5.7.5 is running cleanly in your environment, the user-visible behavior in v5.7.6 is identical — every change in this release is in the *adversarial* path, where v5.7.5 had silent recovery or no defense at all. After v5.7.6 those paths now fail closed and audit loudly.

## Scope

Five tier-1 closures (the ones with documented real-attack-path consequences) plus three tier-1.5 wins that piggy-back on the same release.

| # | Closure | Surface | Disposition |
|---|---|---|---|

## Threat-model rationale

### 1. Downgrade-replay protection

Ed25519 manifest signing answers "is this manifest authentic?" It does not answer "is this manifest the latest?" An adversary who can replay an authentic-but-older manifest (compromised CDN, leaked withdrawn release, one-time signing-key compromise on a single release) could force every client to "update" backwards into a known-vulnerable version. The signed pubkey list and the new HMAC ledger together close that path: every successful manifest verification raises the floor, and a future manifest below the floor is refused even with a valid signature.

### 2. `settings.json` HMAC

Pre-v5.7.6, a local-file attacker (or a buggy backup-restore tool) could rewrite `settings.json` directly. DupeZ would happily load `"kill_switch": false` and proceed without the kill switch. v5.7.6 closes this with the same pattern PatchMonitor and `data_persistence.py` already use: every save also writes `settings.json.hmac` under the per-install DPAPI-sealed `persistence.hmac` secret. Tampered files are quarantined and the loader returns the safe default.

### 3. Audit fail-closed

The audit log is the record of truth for what DupeZ did and when. Pre-v5.7.6, if the hash chain broke, the logger rotated the suspect file aside and silently started a fresh chain. Silent recovery on the integrity-of-truth substrate is the wrong default — if the chain is questionable, refusing to write is safer than continuing under uncertainty. v5.7.6 seals the logger on any chain-break signal, drops a `audit.TAMPERED` sentinel that persists across process restarts, and refuses every subsequent write until the operator runs `dupez --reset-audit`.

### 4. Subprocess hardening

`safe_subprocess.run` and `spawn_detached` are the only sanctioned routes to spawn a child process. v5.7.5 already enforced `shell=False`, absolute-path executables, `CREATE_NO_WINDOW`, audit emit on every spawn and exit, and timeouts on every run. v5.7.6 adds the belt: `STARTUPINFO` with `STARTF_USESHOWWINDOW | SW_HIDE` (any ancillary window the child tries to surface is hidden, not briefly flashed) and `close_fds=True` explicitly on every platform including Windows. The pre-v5.7.6 `close_fds=(os.name != "nt")` opt-out in `spawn_detached` was a Python 3.6-era workaround that no longer applies on modern Python and was leaking parent handles into detached children.

### 5. Webhook host allowlist

v5.7.3 closed `file://` and `ftp://` URLs at sink construction. v5.7.6 also closes `https://attacker.example.com/x` — an off-allowlist HTTPS host is now refused at registration time. Default allowlist: the canonical Discord hosts (`discord.com`, `discordapp.com`, plus `canary.` and `ptb.` for early-deploy operators). Operator-pinned extras live in `app/config/audit_webhook_hosts.json` with an HMAC sidecar so an imported config bundle from an untrusted source cannot widen the allowlist without also owning the install's DPAPI key. Loopback HTTP/HTTPS is unchanged — local dev relays still work.

### 6. `--verify-self`

For frozen PyInstaller builds, the running `dupez.exe` is verified against a sidecar `dupez.exe.sig` that signs the SHA-256 of the executable under the same pinned Ed25519 release key as update manifests. Detects post-install binary tamper (modded builds floated on forums, malware-replaced exes, etc.). Currently an opt-in CLI flag; can be promoted to a startup auto-check once the signing pipeline has shipped enough releases for the sidecar to be ubiquitous on installed bases.

### 7. `--reset-audit`

Operator-facing escape hatch for the v5.7.6 fail-closed audit log. Archives every `audit*.jsonl`, `audit.TAMPERED`, and `audit*.tampered.*` file to `audit-quarantine-<ts>/` inside the audit directory, clears the sentinel, and writes a fresh `audit_chain_reset_by_operator` event as the genesis of the new chain. Designed to be run after the operator has copied the quarantined files off-box for forensic review.

### 8. Cert-pinning infrastructure (audit-only this release)

`app/core/cert_pinning.py` provides SPKI-pinning helpers for the auto-updater's HTTPS calls to GitHub. The `PINS` map is **empty by default in v5.7.6**: every cert chain observed on an update call emits a `cert_chain_observed` audit event with the SPKI hashes seen, so the release engineer can populate the pin set in v5.7.7 with values captured from production rather than guessed from documentation. Shipping known-good pins straight from a paper analysis would brick the updater if any value were off; staged rollout is the right call.

When PINS is non-empty for a host, a chain that matches no pinned SPKI raises `PinViolation` and aborts the connection. The `DUPEZ_DISABLE_CERT_PIN=1` environment variable is the explicit, audit-loud bypass for the recovery path if a future CA rotation breaks the pin set.

## Operator notes

### Migration from v5.7.5

* **`settings.json`:** automatic. On first v5.7.6 load, the existing file is accepted and a sidecar is signed in place. No operator action.
* **Audit chain:** automatic if clean. If the resume code detects a v5.7.5-era unkeyed SHA-384 chain on disk, the existing migration path still runs first; v5.7.6 only fails closed on chains that match neither the current HMAC nor the legacy SHA-384.
* **Webhook URLs:** any operator already using a Discord webhook is unaffected — Discord is in the default allowlist. Operators with custom HTTPS webhook hosts must add the host to `app/config/audit_webhook_hosts.json` before upgrade (or accept the rejection and add it after). The HMAC sidecar for that file can be generated with `python -m app.config.audit_webhook_hosts_sign` (helper to be added in v5.7.7).
* **Update ledger:** initialized to `0.0.0` on first run, advances every successful update. No operator action.

### New CLI flags

```text
dupez --verify-self      Verify dupez.exe against its signed sidecar.
                         Returns exit 0 on match, 3 on tamper, 2 on
                         operational failure. Only meaningful for
                         frozen PyInstaller builds.

dupez --reset-audit      Archive the suspect audit chain to
                         audit-quarantine-<ts>/, clear the
                         audit.TAMPERED sentinel, and resume logging
                         under a fresh HMAC chain. Run only after
                         copying the quarantined files off-box.
```

## Deferred to v5.8.x

ADR-0003 documents four items intentionally deferred because they need design proposals, not mechanical hardening:

* Real Windows AppContainer / Job Object enforcement for plugins (currently declarative).
* DPAPI scope audit (`CRYPTPROTECT_LOCAL_MACHINE = 0` flag verification across every secret site).
* Memory-safe secret handling via `SecretBytes` for ~6 hot paths.
* WER (Windows Error Reporting) opt-out for the elevated helper process via `WerAddExcludedApplication`.
