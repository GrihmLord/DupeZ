# DupeZ v5.7.9

Direct Clumsy ownership, complete user-configurable disruption controls,
resilient startup, secure Smart Ops, and fail-closed release integrity.

> **Release status:** candidate branch only. Do not publish the installer until
> the exact signed GPU, Compat, installer, Defender, updater, frozen-runtime,
> authorized hardware, and manual validation evidence has passed.

## Direct Clumsy integration

- DupeZ now treats the bundled Clumsy process as a first-class engine rather
  than an opaque compatibility fallback. It verifies the Local/Remote capture
  layer, requested module checkboxes, numeric values, module sub-options,
  function preset, trigger mode, and running Start/Stop state before reporting
  success.
- The complete bundled v0.3.4 surface is represented in DupeZ: safe additional
  filter, filter/function preset names, Toggle/Timer, independent per-module
  Inbound/Outbound direction, Lag, Drop, Disconnect, Bandwidth Limiter with
  KB/s or MB/s, Throttle with Drop Throttled, Duplicate, Out of order, Tamper
  with Redo Checksum, Set TCP RST, and the bounded RST-next-packet one-shot.
- The Additional filter defaults to `true`, meaning no extra narrowing. It is
  always ANDed with DupeZ's mandatory exact private-target scope and cannot
  replace or broaden that scope.
- DupeZ owns the exact Clumsy child process it launches. It no longer runs a
  global `taskkill /IM clumsy.exe` sweep, so an unrelated standalone Clumsy
  session is never terminated silently.
- Normal operation launches the PID-owned Clumsy GUI hidden, transparent,
  off-screen, and excluded from normal task switching while its controls remain
  active. The authenticated diagnostic action is the only supported path that
  restores, centers, and focuses the native window.
- Stop first requests Clumsy's Stop state, closes its window, and waits for a
  clean exit. Only the exact owned PID may be killed as a bounded final
  fallback.
- If a standalone Clumsy process is already running, explicit Clumsy mode fails
  with an actionable contention message. Auto mode may use Native WinDivert
  only for effects whose packet semantics remain equivalent.
- GPU/split and Compat/in-process builds use the same direct-manager policy.
  The split build restores diagnostics through the authenticated helper control
  channel; packet contents never cross IPC.

## Engine and event controls

- The AI / Smart Ops view exposes explicit **Auto**, **Clumsy**, and **Native**
  routing for normal DISRUPT actions.
- Capture layer can be automatic or explicitly pinned to **Local / NETWORK** or
  **Remote / NETWORK_FORWARD**. Explicit operator selection survives target
  profile tuning instead of being overwritten silently.
- Users can build an ordered event queue from the current effects. Every event
  has its own enable toggle, engine, capture layer, start delay, duration, and
  halt-or-continue failure policy.
- Event queues run outside the Qt thread and are generation-safe: an old event
  timer cannot stop a newer manual disruption on the same private target.
- Event sequences persist atomically under the per-user DupeZ data directory.
  They do not store target IPs, API keys, or packet contents.
- Native WinDivert remains available for native-only features. Automatic
  fallback is refused whenever switching engines would change requested
  behavior.
- Duplicate count means the same thing on both engines: the UI value is extra
  copies, translated to Clumsy's total-packet count.

## Smart Ops

- Smart Ops provider settings support local Ollama, an OpenAI-compatible HTTPS
  endpoint, or offline rules only.
- API keys are stored through DupeZ's encrypted secrets manager and are never
  written to `settings.json`. The legacy encrypted-key lookup bug that silently
  fell back to a deprecated plaintext field is removed.
- Remote plaintext HTTP provider URLs are rejected; loopback HTTP remains
  available for local Ollama.
- Provider health checks and advisor calls run outside the Qt thread. Newer
  requests cancel stale callbacks so a delayed response cannot overwrite a
  current recommendation or update a destroyed panel.
- Smart recommendations inherit the operator's selected engine and capture
  layer. When a provider is disabled or unavailable, the bounded rule-based
  advisor remains usable.

## Startup and shutdown resilience

- Startup uses an adaptive resource profile to cap Qt worker concurrency and
  reduce thread, stack, and memory pressure on constrained systems.
- The splash busy loop is replaced by a Qt event loop with a bounded watchdog.
  Cooperative cancellation stops between initialization phases, and timeout
  cleanup requests controller and disruption-engine shutdown.
- Chromium map prewarming remains controller-first. Capable systems prewarm
  after recovery succeeds; low-resource systems construct a lightweight,
  retryable map placeholder and initialize QtWebEngine only when the Map tab is
  opened.
- A per-user single-instance mutex prevents duplicate GUI processes from
  colliding over persistence locks and the elevated helper pipe.
- Splash startup preserves the real controller exception and durable log path.
  Incomplete network recovery enters an explicit network-disabled safe mode.
- Dashboard and engine telemetry polling no longer performs helper IPC on the
  Qt UI thread.
- Shutdown is idempotent and stops hotkeys, background monitoring, overlays,
  the disruption manager/helper, controller, and Qt workers.

## Release integrity

- PyInstaller builds recreate an isolated `.build-venv`, clear Python path
  injection variables, and install build/runtime dependencies from separate
  hash-pinned locks.
- The runtime lock verifier checks that every direct requirement has a
  compatible exact pin, preserved platform markers, and SHA-256 hashes. CI does
  not silently upgrade broad ranges merely because a new upstream version was
  published.
- Every executable is inspected before signing across both PyInstaller archive
  layers. The v5.7.9 gate rejects unfinished Group Finder packages and optional
  voice/scientific stacks outside the production lock.
- Packaged config, icon, and theme files come from an explicit manifest;
  generated HMACs, Python source/bytecode, and untracked local files cannot leak
  into the executable.
- Pull requests build both unsigned GPU and Compat variants, execute their
  runtime-import verifier, and rerun frozen archive policy before release
  signing begins.
- The protected release host requires the real Authenticode certificate, the
  existing updater Ed25519 private key, Inno Setup, SignTool, current Defender,
  valid SHA-256/RFC3161 signatures, SBOM/VEX/provenance, updater sidecars, and
  final distribution preflight. The protected workflow cannot publish a GitHub
  release.
- GPU and Compat signed files must pass architecture-correct normal and
  forced-low-resource GUI lifecycle tests. Draft staging additionally requires
  a hash-bound operator attestation for diagnostics, map retry, scan,
  disruption, queues, hotkeys, tray, overlay, account tracking, Smart Ops,
  support bundle, and installer fresh/upgrade/uninstall behavior.
- Release staging publishes only a sanitized validation attestation. Raw host
  and operator evidence remains private on the release machine.
- Previous update-manifest sidecars are removed before each variant build.
  Release preflight verifies the pinned Ed25519 signature and confirms manifest
  version, installer filename, size, and SHA-256 against exact
  `DupeZ_Setup.exe`.
- Bundled `clumsy.exe` is byte-identical to the credited
  [Kalirenegade v0.3.4 release](https://github.com/kalirenegade-dev/clumsy/releases/tag/0.3.4).
  The provenance record pins source commit
  `bc87e73066168520d76122c9165e99ea703b166c` and the source archive hash.

## Validation status

Automated source validation covers:

- Windows Python 3.11 and 3.12 complete pytest suites;
- AST parsing, secret scanning, dependency review, and CodeQL;
- deterministic hashed-lock verification;
- GPU and Compat PyInstaller builds, runtime imports, and frozen archive policy;
- Windows PowerShell parsing and static policy checks for the installer,
  signing/Defender finalizer, architecture-correct frozen validator, manual
  attestation recorder, and draft/publication staging scripts.

Authorized source-mode PS5 evidence completed the full bundled-Clumsy control
matrix on the approved private IP/MAC, including every module, direction,
sub-option, numeric field, Timer mode, function preset, diagnostic restoration,
RST one-shot, owned cleanup, no orphan process, and clean repository restoration.
The final exact-head hidden/no-flash hardware gate remains required before PR
merge.

Hardware tests remain opt-in and require Administrator access plus an explicitly
authorized private-network IP and MAC. They never select the first scanned
device.

Required environment:

```text
DUPEZ_RUN_HARDWARE_SMOKETEST=1
DUPEZ_SMOKETEST_TARGET_IP=192.168.137.42
DUPEZ_SMOKETEST_TARGET_MAC=00:11:22:33:44:55
```

## Upgrade

After every signing, Defender, updater, frozen lifecycle, installer, manual, and
authorized hardware gate passes, install `DupeZ_v5.7.9_Setup.exe` or use stable
`DupeZ_Setup.exe` from the latest GitHub release. Until then, v5.7.9 is a release
candidate and must not replace the current public release.
