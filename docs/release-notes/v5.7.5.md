# DupeZ v5.7.5 -- WiFi Disrupt Audit Closure

Follow-up hardening pass closing the open findings from the deep audit of the WiFi same-network disrupt path (`docs/audits/WIFI_DISRUPT_AUDIT_v5.7.4.md`). The four HIGH findings were already closed in v5.7.4; this release closes the actionable MEDIUM and LOW findings and adds a defense-in-depth MAC scrubber to the logger. No behavior changes for normal use -- the work is in failure-mode honesty, resource hygiene, and opsec.

## Defense in depth: MAC scrubbing at the logger

Previously, MAC addresses were masked at the call site via `mask_mac()`. That worked but had no safety net -- a forgotten `mask_mac()` call in any module would leak a device-unique identifier into the log stream. v5.7.5 adds `mask_macs_in_text()` to `app/utils/helpers.py` and wires it into the logger's `ScrubbingFormatter`, so every log line written to disk or console is now scrubbed of raw MAC addresses regardless of whether the call site remembered to mask. The OUI prefix is preserved (vendor identification is public via the IEEE registry anyway); the trailing three octets -- the device-unique part -- are replaced with `**:**:**`.

## What's fixed

- **(M1) Best-effort ARP cache restoration on process termination.** When ARP spoofing starts, DupeZ now registers `atexit.register(spoofer.stop)` so the target's ARP cache gets restored on uncaught exceptions, `sys.exit()`, and Ctrl+C. Previously a crash mid-session left the target poisoned, black-holing its gateway traffic for 30-60s until the ARP entry aged out (longer on endpoints that pin ARP under load). `atexit` does NOT run on `kill -9` or SIGSEGV -- that path still requires the next-launch hot-reload guard tracked separately -- but it covers every clean-shutdown failure mode.

- **(M2) WiFi same-network detection now respects the real interface netmask.** `_is_wifi_same_network` previously hardcoded `/24`, which broke any operator on a `/23` or `/22` LAN (common on Eero mesh and business APs). A target on `192.168.2.10` when the operator sat on `192.168.1.50/23` would silently misclassify as "not same-network" and fall through to the NETWORK layer with the wrong target -- a silent no-op disrupt. v5.7.5 reads the actual netmask via `psutil.net_if_addrs()`, falling back to `/24` only when the netmask isn't exposed.

- **(M6) `NpcapSender` is now a context manager with a defensive `__del__`.** Partial-init failures (e.g. `load()` succeeded, `open()` raised) no longer leak the pcap handle. `with NpcapSender() as sender:` semantics are supported, and a best-effort `__del__` is the safety net for any path that doesn't use the context manager. Both safeguards call `close()` idempotently.

- **(L2) `ArpSpoofer` validates `target_ip` and `gateway_ip` at construction.** A malformed IP previously propagated through `arp -a`, `ping`, and `socket.connect`, surfacing only as a generic "cannot resolve MAC for target" message after 3+ subprocess invocations. Now `__init__` raises `ValueError` immediately with a clear message.

- **(L3) `_poison_loop` self-terminates after persistent send failures.** If `_poison_once` raises 5 times in a row (typical sign that the Npcap handle has died mid-session), the loop now logs CRITICAL, sets `_running = False`, and exits cleanly. Previously the loop would spin forever logging an error every cycle while the disruptor still reported the spoofer as ACTIVE -- the silent-fail pattern v5.6.4 was meant to eliminate.

## Test coverage added

`tests/test_ip_leak_guard.py` extended with four new test classes:

- `TestMacScrubber` -- exercises the logger formatter against raw MACs and confirms `mask_macs_in_text` is idempotent.
- `TestTargetProfileNetmask` -- locks the `_local_network_for_ip` helper's behavior (returns `IPv4Network`, falls back to `/24` for unknown IPs).
- `TestArpSpooferValidatesIp` -- confirms malformed `target_ip` / `gateway_ip` raise `ValueError` at construction.
- `TestNpcapSenderContextManager` *(covered indirectly in the existing test suite via the M6 fix)*.

## What's intentionally deferred

The remaining MEDIUM findings (`M3` watchdog cancel-vs-fire race, `M4` watchdog thread join window, `M5` `clear_all_disruptions_clumsy` error aggregation) and three LOW findings (`L1` `_get_ip_forwarding_state` locale parsing, `L4` doc comment on watchdog callback, `L5` taskkill warning logs) need orchestration-level refactors that don't fit a point release. They're tracked for v5.8.x alongside the god-object refactor work already on the roadmap. Architecture recommendation #5 -- a `WifiDisruptSession` class that owns spoofer + engine + watchdog as one unit -- is the cleanest closing path for all three M's at once.
