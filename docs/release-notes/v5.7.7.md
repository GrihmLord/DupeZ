# DupeZ v5.7.7 — Defensive hardening and release polish

v5.7.7 is a safety, reliability, and release-discipline release. It does not add
new competitive-game capability. The supported boundary is owned-lab diagnostics:
private DayZ servers, local devices the operator controls, explicit
authorization, short automatic deadlines, dry-run previews, auditable rollback,
and redacted support artifacts.

## What changed

- Public method allowlists now expose only bounded diagnostic impairments such
  as lag, loss, throttling, reordering, corruption testing, temporary
  disconnect, and statistical network models.
- Legacy high-risk timing/pulse method names are no longer selectable through
  validation, preset editing, Smart Mode, or the local advisor.
- Active DayZ presets, tips, GPC generation, README, roadmap, and audit docs
  were scrubbed to describe connection diagnostics rather than game-state
  outcomes.
- Release metadata is synchronized across runtime version, build scripts,
  installer metadata, Windows manifests, README, and release notes.
- The release gate now covers active-content safety, source-leak scanning,
  binary provenance, release preflight, privacy inventory, storage migration,
  support-bundle redaction, Pktmon planning, network health, performance smoke,
  ruff, compile, and pytest.

## Operator impact

- Existing safe presets continue to load.
- Presets or advisor prompts that relied on quarantined legacy method names are
  rejected or mapped to bounded diagnostic alternatives.
- Active CLI operations require the owned/authorized-network acknowledgement
  unless `--dry-run` is used.
- Support bundles remain local files and redact raw targets, secrets, packet
  payloads, user-specific filesystem roots, account contents, IPs, and MACs.
