# DupeZ v5.6.7 — Account-Tracker XLSX Fix + Multi-Script-Device Support

Two operator-facing bugs in one release.

## Fixed: account-tracker silently skipping rows

The XLSX importer auto-detected the header row and matched 8 of 9 columns (EMAIL, LOCATION, STATUS, STATION, GEAR, HOLDING, LOADOUT, NEEDS) but left column A unmapped because the header cell above the account names was blank. Then every data row was skipped because `row_data['account']` was always empty. Workbooks with this layout — extremely common when the account name acts as the row label — silently imported 0/N accounts.

v5.6.7 adds an account-column inference step: when no column maps to "account" but an unmapped leading column has consistent text content in the data rows, that column is adopted as the account column. Confirmed on Grihm's actual workbook (11/11 accounts imported, 0 skipped — was 0/11 on v5.6.6).

## Added: Titan One / Titan Two support in the game-script panel

The panel was Cronus-only. Titan One and Titan Two from ConsoleTuner compile the same .gpc syntax (Cronus-derived dialect), but their devices weren't detected and exports went to the wrong library folder.

- **Detection** — `SUPPORTED_VIDS = ("2508", "2F0A")` now covers Cronus (legacy) and Titan. USB description string disambiguates Cronus Zen / Cronus Max / Titan One / Titan Two.
- **Library finding** — new `find_gtuner_library()` locates Gtuner IV / Gtuner II / ConsoleTuner script folders; `find_cronusmax_library()` handles the legacy CronusMAX Plus IDE.
- **Export routing** — `get_default_export_path(device_type)` picks Zen Studio for Cronus, Gtuner for Titan, CronusMAX Plus for the legacy MAX. Falls back to `Documents/DupeZ/GPC` for unknown/unconnected. Backward compat: callers passing no argument get the pre-v5.6.7 Zen-first behavior.
- **UI** — panel renamed "GPC / CRONUS" → "GAME SCRIPTS"; device-status banner shows human label ("Cronus Zen", "Titan Two") instead of the raw `device_type` tag.
- **Backward compat** — `CronusDevice` is now an alias for the new `ScriptDevice` dataclass. Existing code keeps working.

## What didn't change

- Script content: `GPCGenerator` emits the same .gpc bytes. Operators with both ecosystems installed can produce one script and run it on either device class.
- Cronus-only workflows: zero behavioral change. If you only have a Cronus Zen and Zen Studio installed, EXPORT goes exactly where it did before.

## Auto-update note

If you're on v5.6.6, auto-update will work automatically (the signed manifest is on this release). If you're on v5.6.5 or older — manual download required once; future updates will install automatically thereafter.
